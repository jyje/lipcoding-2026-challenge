# 03. Implementation Timeline (3시간)

## 0) 전체 원칙

- 먼저 끝나는 기능부터: 분석 1회 -> 결과 렌더 -> 배포
- 기능 추가보다 실패 방지 우선
- 매 30분마다 통합 점검

## 1) 시간표 (분 단위)

### T+00 ~ T+15

- 저장소/런타임 준비
- 환경변수 템플릿 작성
- 공통 스키마 파일 생성 (`Action`, `AnalyzeResponse`)

완료 기준

- FE/BE 실행 커맨드 모두 동작

### T+15 ~ T+45

BE 트랙

- `/api/analyze` 라우트 생성
- SDK 호출 래퍼 + 스키마 검증 + fallback

FE 트랙

- 입력 화면 + 분석 버튼 + 로딩/에러 상태
- 목 응답 렌더링

완료 기준

- 목 데이터 기준으로 결과 화면 완성

### T+45 ~ T+75

BE 트랙

- `/api/actions`, `/api/actions/:id` 구현 (메모리 or SQLite)
- 최소 단위 테스트 (happy path)

FE 트랙

- Top 3 카드, 플랜 블록 UI
- 완료 토글 연결

완료 기준

- 실제 API 연결 후 액션 완료 토글 동작

### T+75 ~ T+105

- `/api/replan` 구현
- 남은 시간 입력 -> 재분석 결과 반영
- 구조화 로그 추가

완료 기준

- 재계획 시나리오 1회 성공

### T+105 ~ T+135

- Azure 배포 (BE -> FE)
- 환경변수 주입
- CORS/기본 헬스체크 확인

완료 기준

- 배포 URL에서 analyze end-to-end 동작

### T+135 ~ T+165

- 데모 데이터 세트 준비
- 데모 스크립트 리허설 2회
- 장애 대비 플랜 B 확인

완료 기준

- 60초 시연 무중단 2회 연속 성공

### T+165 ~ T+180

- README 정리
- 알려진 제한사항 정리
- 최종 제출

## 2) 실패 방지 게이트

- Gate 1 (T+45): analyze API 응답 확인 못 하면 UI 기능 확장 중단
- Gate 2 (T+105): 배포 안 되면 기능 추가 중단, 배포에만 집중
- Gate 3 (T+150): 데모가 끊기면 로컬 백업 데모 루트 준비

## 3) 담당 분리 예시

- FE 담당: 입력/결과/토글/상태 UX
- BE 담당: 분석/재계획/API/로그
- 통합 담당(겸임): 타입 계약 + 배포 + 데모 리허설
